# 949Fantasy Metrics Glossary


> **Canonical source:** `/Users/matthewhanratty/Documents/New project/949fantasy-metrics-glossary.md`


## Purpose

This document defines the first-pass 949Fantasy metric system.

The goal is to give Cursor and Claude consistent metric names, definitions, formulas, and data requirements. These definitions should evolve as the model is tested, but the product should use this vocabulary from the start.

## Core Metric Families

949Fantasy metrics fall into eight groups:

- Team pacing metrics.
- Position group metrics.
- Player projection metrics.
- Player value metrics.
- Floor/ceiling and uncertainty metrics.
- League comparison metrics.
- Draft value metrics.
- Context/condition advantage metrics.
- Lineup optimization metrics.
- Ranking and tape-analysis metrics.

## Foundational Terms

### Actual Points

The fantasy points a player or team actually scored for a completed week.

Depends on league scoring format:

- Standard.
- Half PPR.
- PPR.
- Future: custom league scoring.

### Projected Points

The expected fantasy points for a player, position group, or team before the week is played.

Source may come from:

- 949 model.
- SportsDataIO projections.
- Temporary RapidAPI-derived model.
- Imported platform projection, if available.

### Expected Points

The season or weekly target point output used for pacing analysis.

For v1, expected points can be derived from projected points.

```txt
expected_points = sum(projected_points)
```

### Position Group

A fantasy roster grouping:

- QB
- RB
- WR
- TE
- FLEX
- DST
- K

FLEX should be treated carefully because it can include different real positions depending on league settings.

### Scope

Many metrics must be stored with an explicit comparison scope.

Common scopes:

- Team.
- Opponent.
- Bench.
- League starters.
- League full rosters.
- Waiver wire.
- All NFL players at position.

## Team Pacing Metrics

### Team Points Per Week

The user's actual weekly team score.

```txt
team_points_week = sum(actual_points for starting lineup in week)
```

Used on:

- Snapshot.
- Metrics Overview.
- Projections.

### Expected Team Points Per Week

The user's projected weekly team score.

```txt
expected_team_points_week = sum(projected_points for starting lineup in week)
```

### Season Expected Points

The user's total expected points across the season.

```txt
season_expected_points = sum(expected_team_points_week for all regular season weeks)
```

For future weeks, use projected lineups or best available roster projections.

### Season Actual Points

The user's total actual points across completed weeks.

```txt
season_actual_points = sum(team_points_week for completed weeks)
```

### Points Pace

The team's current per-week scoring pace.

```txt
points_pace = season_actual_points / completed_weeks
```

### Projected Season Finish

The projected final season points if current pace continues plus future projections.

Two possible versions:

Current pace version:

```txt
projected_season_finish = points_pace * total_regular_season_weeks
```

Hybrid version:

```txt
projected_season_finish = actual_points_completed_weeks + projected_points_future_weeks
```

Use the hybrid version when reliable future projections exist.

### Team Projection Variance

Percent difference between actual points and expected points.

```txt
team_projection_variance_pct =
  (season_actual_points - expected_points_completed_weeks)
  / expected_points_completed_weeks
```

Display:

- Green if positive.
- Red if negative.

## Position Group Metrics

### Position Actual Points

Actual points generated by a position group.

```txt
position_actual_points =
  sum(actual_points for lineup slots in position group)
```

Example:

All starting RB slots for a team across a selected time range.

### Position Expected Points

Expected points for a position group.

```txt
position_expected_points =
  sum(projected_points for lineup slots in position group)
```

### Position Points Per Week

Average weekly points from a position group.

```txt
position_points_per_week =
  position_actual_points / completed_weeks
```

### Position Expected Points Per Week

Expected weekly points from a position group.

```txt
position_expected_points_per_week =
  position_expected_points / number_of_weeks_in_scope
```

### Position Variance Percent

How far a position group is above or below expectation.

```txt
position_variance_pct =
  (position_points_per_week - position_expected_points_per_week)
  / position_expected_points_per_week
```

Example:

If RB expected output is `17.6` points per week and actual output is `16.5`:

```txt
(16.5 - 17.6) / 17.6 = -6.25%
```

### Position League Rank

How a user's position group ranked against other teams in the league for a week.

```txt
position_league_rank =
  rank(user_position_points_week among all teams' same position slots)
```

Example:

QB was #2 of 12 starting QBs in the league for Week 4.

### Position Field Percentile

How a player or position group performed against all players at that position, including waiver-wire players.

```txt
position_field_percentile =
  percentile_rank(actual_points among all players at same position for week)
```

Used for League heatmap.

## Player Projection Metrics

### Player Projected Points

Expected fantasy points for a player in a given week.

```txt
player_projected_points_week = model projection for player/week/scoring_format
```

### Player Actual Points

Actual fantasy points for a player in a given week.

```txt
player_actual_points_week = scoring_formula(player_weekly_stats)
```

### Player Projection Error

Raw difference between actual and projected points.

```txt
projection_error = actual_points - projected_points
```

### Player Projection Error Percent

Percent difference between actual and projected points.

```txt
projection_error_pct = (actual_points - projected_points) / projected_points
```

Guardrail:

- If projected points is 0, return null/unavailable.

### Player Projection Accuracy

One possible user-friendly score for projection closeness.

```txt
projection_accuracy =
  max(0, 1 - abs(actual_points - projected_points) / max(projected_points, 1))
```

Display as percent.

This is mainly for internal validation and user-facing confidence later.

### Weekly Projection Delta

How much a player's current weekly projection moved from the last projection snapshot.

```txt
weekly_projection_delta =
  current_projected_points_week - previous_projected_points_week
```

This should be visible on player pages and rankings when meaningful.

## Player Value Metrics

### Position Baseline Points

The average or median point output for a position group.

The roadmap describes value as player points divided by position group average.

Recommended v1 baseline:

```txt
position_baseline_points =
  average(season_points for relevant players at position)
```

Alternative for stability:

```txt
position_baseline_points =
  median(season_points for startable/relevant players at position)
```

Decision needed:

- Average is closer to the original roadmap.
- Median is more robust against outliers.

### Player Value Score

The player's scoring power relative to the position baseline.

```txt
player_value_score =
  player_points / position_baseline_points
```

Example:

If QBs average `315` points and a QB scores `385`:

```txt
385 / 315 = 1.22
```

Interpretation:

- `1.00` = position baseline.
- `1.20` = 20% above baseline.
- `0.80` = 20% below baseline.

### Three-Year Player Value

The player's multi-year value against recent position baselines.

```txt
three_year_player_value =
  average(player_points_last_3_years)
  / average(position_baseline_points_last_3_years)
```

Use this for draft research and player profile stability.

### Per-Game Value Score

The player's points-per-game scoring power relative to position average points per game.

```txt
per_game_value_score =
  player_points_per_game / position_baseline_points_per_game
```

This is important because injured players can have low season totals but strong weekly value.

### Value Stability Score

How consistent a player's value score has been across seasons or weeks.

Initial version:

```txt
value_stability_score =
  1 - coefficient_of_variation(player_value_scores_over_time)
```

Clamp between 0 and 1.

Purpose:

- Separate steady value players from volatile value players.

## Floor, Ceiling, And Uncertainty Metrics

### Floor Points

A low-end expected outcome for a player or lineup.

Recommended v1:

```txt
floor_points = 20th percentile projected outcome
```

Alternative:

Use 25th percentile for simpler communication.

### Median Points

The most central expected outcome.

```txt
median_points = 50th percentile projected outcome
```

### Ceiling Points

A high-end expected outcome.

Recommended v1:

```txt
ceiling_points = 80th percentile projected outcome
```

Alternative:

Use 75th or 95th percentile depending how aggressive the product wants to be.

### Floor/Ceiling Hit

Whether actual points landed inside the projected floor/ceiling range.

```txt
floor_ceiling_hit =
  actual_points >= floor_points
  and actual_points <= ceiling_points
```

### Floor/Ceiling Hit Rate

How often a player's actual points landed inside the modeled range.

```txt
floor_ceiling_hit_rate =
  count(floor_ceiling_hit) / count(eligible_games)
```

Roadmap goal:

- At least 70% hit rate.
- Exclude games where meaningful injury context makes the player miss most or all of the game.

### Projection Range Width

How wide the floor/ceiling range is.

```txt
projection_range_width = ceiling_points - floor_points
```

### Volatility Score

How unpredictable a player is week to week.

Initial version:

```txt
volatility_score =
  standard_deviation(player_weekly_points) / average(player_weekly_points)
```

Higher = more volatile.

### Confidence Score

A user-facing confidence score for a projection.

Initial version should combine:

- Recent sample size.
- Role stability.
- Injury status.
- Projection range width.
- Historical volatility.

Placeholder formula:

```txt
confidence_score =
  weighted_score(sample_size, role_stability, health_status, inverse_volatility, inverse_range_width)
```

This should be refined after model testing.

### Low-Sample Flag

A guardrail metric for condition or split-driven claims.

```txt
low_sample_flag = sample_size < minimum_required_sample
```

Use this to suppress or soften red/yellow/green condition claims when history is too thin.

## Boom/Bust Metrics

Draft Rankings should display this column as `Boom / Bust`. A row like `51 / 16` means:

- `51%` chance the player lands in the boom outcome band.
- `16%` chance the player lands in the bust outcome band.
- `33%` neutral or expected outcome range.

The three bands should sum to 100% conceptually, though the UI only displays the boom and bust numbers.

### Boom Probability

Probability that a player significantly exceeds expectation.

Initial definition:

```txt
boom_probability =
  probability(actual_points >= projected_points * 1.20)
```

Alternative:

Use position-specific boom thresholds.

Examples:

- QB boom: 25+ points.
- RB/WR boom: 20+ points.
- TE boom: 15+ points.

### Bust Probability

Probability that a player significantly underperforms expectation.

Initial definition:

```txt
bust_probability =
  probability(actual_points <= projected_points * 0.80)
```

Alternative:

Use position-specific bust thresholds.

### Neutral / Expected Probability

Probability that a player lands between the boom and bust thresholds.

Initial definition:

```txt
neutral_probability =
  max(0, 1 - boom_probability - bust_probability)
```

This value is implied in Draft Rankings tooltips, not shown as a separate column.

### Risk Label

Human-readable risk classification.

Inputs:

- Bust probability.
- Volatility score.
- Injury status.
- Role stability.
- Matchup grade.

Initial labels:

- Low.
- Medium.
- High.

Example:

```txt
if bust_probability < 0.20 and volatility_score is low -> Low
if bust_probability between 0.20 and 0.35 -> Medium
if bust_probability > 0.35 or injury concern exists -> High
```

## League Comparison Metrics

### Your Players vs Field

Compares the user's players at a selected position against the average of all players at that position.

```txt
your_players_vs_field_week =
  user_position_points_week - all_players_position_average_week
```

Display as raw points and percent:

```txt
your_players_vs_field_pct =
  your_players_vs_field_week / all_players_position_average_week
```

### Team vs Opponent Position Edge

Compares a user's lineup slot or position group against the weekly opponent.

```txt
position_edge_vs_opponent =
  user_position_points_week - opponent_position_points_week
```

Used for Team heatmap.

### Starter vs Bench Edge

Compares a starter against a comparable bench player at the same position.

```txt
starter_vs_bench_edge =
  starter_points - best_or_average_bench_same_position_points
```

Decision needed:

- Compare against best bench option.
- Or compare against average bench option.

Recommendation:

- Use best bench option for start/sit regret analysis.

### League Average Points

Average points scored by starting lineups in the league.

```txt
league_average_points_week =
  average(team_points_week across all league teams)
```

### League Average Difference

User team scoring compared to league average.

```txt
league_average_diff =
  user_team_points_week - league_average_points_week
```

```txt
league_average_diff_pct =
  league_average_diff / league_average_points_week
```

### Waiver Wire Edge

Compares the user's player or position group against relevant waiver-wire alternatives.

```txt
waiver_wire_edge =
  user_player_or_group_points - waiver_wire_baseline_points
```

This can be used in field-comparison views and start/sit opportunity surfaces.

## Draft Value Metrics

### ADP

Average draft position.

```txt
adp = average draft pick number across source drafts
```

Sources may include:

- SportsDataIO.
- FantasyPros, if licensed.
- Platform-specific draft data.

### Draft Round

The round in which a player was selected.

```txt
draft_round = ceil(pick_number / league_team_count)
```

### Drafted Player Points

Season-to-date or full-season points from a user's drafted player.

```txt
drafted_player_points = player_actual_points_season_to_date
```

### Best In Round Points

Highest points from any player drafted in the same round.

```txt
best_in_round_points =
  max(points for players drafted in same round)
```

### Best At Position Points

Highest points from any player in the drafted player's position group.

```txt
best_at_position_points =
  max(points for players at same fantasy position)
```

### Draft Pick Value Return

How much value the user's pick returned relative to the best player in that round.

```txt
draft_pick_value_return =
  drafted_player_points / best_in_round_points
```

### Position Opportunity Cost

How much value the user lost or gained relative to the best player at that position.

```txt
position_opportunity_cost =
  drafted_player_points - best_at_position_points
```

Usually negative unless the user has the top player at that position.

### ADP Value Delta

Difference between a player's current rank and ADP-implied rank.

```txt
adp_value_delta =
  adp_rank - current_overall_rank
```

Interpretation:

- Positive = player is outperforming draft cost.
- Negative = player is underperforming draft cost.

### Draft Availability Probability

Probability a player will be available at a future pick.

Inputs:

- ADP.
- Current draft state.
- League size.
- Pick number.
- Position demand.
- Players already selected.

Placeholder:

```txt
draft_availability_probability =
  model_probability(player_available_at_pick)
```

This likely requires simulation.

### Draft Score

Composite score for the current pick context.

```txt
draft_score =
  projected_value_score
  + market_discount_score
  + value_over_next_available
  + scarcity_score
  + tier_cliff_score
  + roster_fit_score
  + upside_score
  + platform_edge_score
  - injury_risk_penalty
  - overpay_penalty
  - roster_fragility_penalty
```

Use this as the main GM ranking for the current pick. Store component scores separately for explanations and validation.

### Market Discount

How much cheaper a player is in the draft market than the 949 model believes he should be.

```txt
market_discount =
  nine49_projected_value
  - adp_implied_market_price
```

Positive values indicate a potential discount. The exact scale should be normalized before it is combined into `draft_score`.

### Platform Edge

Difference between where a platform presents a player and where 949 ranks the player.

```txt
platform_edge = platform_rank - nine49_rank
```

Positive values mean the platform is likely showing the player lower than 949 values him, which can create a draft-room discount.

### Value Over Next Available

The value of taking a player now instead of waiting until the user's next pick.

```txt
value_over_next_available =
  current_player_value
  - expected_best_available_same_position_at_next_user_pick
```

This is especially important in snake drafts because it captures positional cliffs and likely future scarcity.

### Smoothed Next Available Value

Expected value of the best same-position option at the user's next pick, smoothed across a pick window instead of using a single ADP cutoff.

```txt
smoothed_next_available_value =
  weighted_average(projected_value of same-position candidates near next_user_pick_window)
```

The smoothing window should grow later in the draft because uncertainty increases.

### Snake Value

Snake-draft-adjusted value that combines baseline value with a controlled VONA/opportunity-cost adjustment.

```txt
snake_value =
  value_over_baseline
  + opportunity_cost_weight * value_over_next_available
```

Use this as the main scarcity-aware value input for snake drafts.

### Tier Drop-Off Slope

Measures how much value is about to disappear at a position.

```txt
tier_dropoff_slope =
  current_tier_average_value
  - next_tier_average_value
```

High tier drop-off slope should increase urgency when the user may not pick again before the tier is depleted.

### Spend Efficiency

Expected return on draft capital.

```txt
spend_efficiency =
  expected_player_value / pick_capital_cost
```

Use this to grade whether a player is a good use of the current pick.

### Scarcity Pressure

How likely a position is to dry up before the user picks again.

```txt
scarcity_pressure =
  expected_position_demand_before_next_pick
  * tier_dropoff_slope
  / max(remaining_startable_players_at_position, 1)
```

Higher values indicate that waiting at the position may be expensive.

### Source Reliability Score

Position-specific trust weight for projections, ADP, 949 ranks, and platform rank.

```txt
source_reliability_score =
  weighted_score(source_accuracy_by_position, recency, sample_size)
```

Use backtesting to update this over time. Do not assume ADP or projections are always better across every position.

### Platform Visibility Score

How visible or hidden a player is in the user's draft room.

```txt
platform_visibility_score =
  platform_rank - nine49_rank
```

Higher values mean the player may be buried on the platform board relative to 949's valuation.

### Roster Portfolio Score

Measures whether the current roster is balanced, fragile, upside-heavy, or floor-heavy.

Initial inputs:

- Starting slot coverage.
- Bench depth by position.
- Injury risk concentration.
- Bye-week concentration.
- Floor/ceiling mix.
- Positional scarcity exposure.

## Game Context Metrics

### Condition Split Score

How a player historically performs in a specific game condition.

Conditions:

- Indoor/outdoor.
- Weather.
- Day of week.
- Game time.
- Home/away.

Initial definition:

```txt
condition_split_score =
  average(points in condition) / average(points in all games)
```

### Condition Tier

Green/yellow/red dot classification.

Within a player's own historical games:

- Green = top third.
- Yellow = middle third.
- Red = bottom third.
- Gray = neutral/unavailable/not applicable.

```txt
condition_tier =
  tercile(condition_split_score or points in condition)
```

This metric must respect the `low_sample_flag`.

### Game Environment Score

Summary score for a player's upcoming game conditions.

Inputs:

- Stadium.
- Weather.
- Time.
- Day.
- Home/away.
- Opponent.

Placeholder:

```txt
game_environment_score =
  weighted_score(stadium_tier, weather_tier, time_tier, day_tier, home_away_tier)
```

### Venue Type

Normalized venue classification for a game.

Values:

- Indoor.
- Outdoor.
- Retractable roof, open.
- Retractable roof, closed.

This supports both condition analysis and player page context.

## Matchup Metrics

### Matchup Grade

A simple grade for player matchup quality.

Inputs:

- Opponent points allowed to position.
- Defensive rank.
- Vegas/implied team total if odds are available.
- Pace/play volume.
- Weather.
- Player role.

Initial display:

- Great.
- Good.
- Neutral.
- Tough.
- Bad.

### Opponent Position Allowance

How many fantasy points the opponent allows to a position.

```txt
opponent_position_allowance =
  average(points_allowed_to_position_by_opponent)
```

### Matchup Edge

Player projection adjusted by opponent position allowance.

```txt
matchup_edge =
  player_projected_points - position_average_projection_against_opponent
```

## Ranking Metrics

### Weekly Rank

Player rank for the current week within selected scope.

Scopes:

- Overall.
- QB.
- RB.
- WR.
- TE.
- FLEX.
- DST.
- K.

### Rest-Of-Season Rank

Player rank based on projected remaining season points and value.

```txt
ros_points = sum(projected_points for remaining weeks)
```

### Weekly Rank Delta

Change from previous rank snapshot.

```txt
weekly_rank_delta =
  previous_rank - current_rank
```

Positive means the player moved up.

### Rank Preview Depth

How much of the ranking table is visible to a non-premium user.

Example:

```txt
rank_preview_depth = 10
```

This is a product metric/configuration rather than a football metric, but it should be explicit because the Rankings page has a preview wall requirement.

### Tier

Group of players with similar expected value.

Initial tiering options:

- Manual analyst tier.
- Projection gap-based tier.
- Cluster-based tier.

V1 can use projection gap-based tiers:

```txt
new_tier when projected_points gap exceeds threshold
```

## Start/Sit Metrics

### Start Confidence

How strongly 949 recommends starting a player.

Inputs:

- Projected points.
- Floor.
- Ceiling.
- Risk label.
- Matchup grade.
- Alternatives on roster.

Display:

- Start.
- Lean Start.
- Toss-Up.
- Lean Sit.
- Sit.

### Optimized Lineup Projection

The projected total points for the best legal lineup given the roster and league settings.

```txt
optimized_lineup_projection =
  max(sum(projected_points for legal lineup combinations))
```

### Lineup Lift

Expected gain from optimized lineup over current lineup.

```txt
lineup_lift =
  optimized_lineup_projected_points - current_lineup_projected_points
```

### Regret Points

Points lost by starting a lower-scoring player over a better bench option.

```txt
regret_points =
  best_bench_option_actual_points - starter_actual_points
```

Only positive regret should usually be shown.

### Lineup Floor

Low-end expected outcome for the selected lineup.

```txt
lineup_floor = sum(player_floor_points for selected lineup)
```

### Lineup Median

Median expected outcome for the selected lineup.

```txt
lineup_median = sum(player_median_points for selected lineup)
```

### Lineup Ceiling

High-end expected outcome for the selected lineup.

```txt
lineup_ceiling = sum(player_ceiling_points for selected lineup)
```

## Tape Metrics

### Player Tape Range

The visible weekly range for a player on the tape view.

```txt
player_tape_range = [floor_points, ceiling_points]
```

### Tape Miss Distance

How far above or below the projection range a player's actual score landed.

```txt
tape_miss_distance =
  0 if floor_points <= actual_points <= ceiling_points
  floor_points - actual_points if actual_points < floor_points
  actual_points - ceiling_points if actual_points > ceiling_points
```

### Tape Direction

Whether the player beat projection, missed projection, or landed near expectation.

Values:

- Beat.
- Inline.
- Missed.

## Validation Metrics

### Mean Absolute Error

Average absolute projection miss.

```txt
mae = average(abs(actual_points - projected_points))
```

### Bias

Whether projections are systematically high or low.

```txt
bias = average(projected_points - actual_points)
```

Positive means model overprojects.

### Hit Rate Over Projection

How often a player beats projection.

```txt
over_projection_rate =
  count(actual_points > projected_points) / count(eligible_games)
```

### Floor/Ceiling Calibration

Whether the projection range is too wide, too narrow, or accurate.

Core target:

```txt
floor_ceiling_hit_rate >= 70%
```
